# GoCMS Onboarding 101

Этот гайд — практическая посадка в GoCMS для первого знакомства.
Цель: за 20–40 минут понять, как поднять систему, добавить первые сущности,
организовать безопасность и получить базовую эксплуатационную готовность.

Документ ориентирован на работу с проектом как с платформой CMS с выделенным админ-панелью,
REST/GraphQL интерфейсами и плагинной моделью.

## 1. Что такое GoCMS на практике

GoCMS — модульная платформа: **ядро с расширениями**, **Admin UI** (`/go-admin`, `/go-login`, `/go-logout`), **REST** (`/go-json`), **публичный рендер** по опубликованным данным. Архитектуру описывают три слоя документации/кода: `go-codex/en` (внешний контракт), `go-stack/en` (Go-архитектура), `go-ui8kit/en` (профиль админского UI).

**Подробнее:** [что такое GoCMS на практике](./architecture/what-is-gocms-in-practice.md) · [оглавление раздела «Архитектура»](./architecture/README.md).

## 2. Как быстро начать: что нужно выбрать первым

Перед первым запуском определите три вещи:

1. **Runtime профиль**
  - `admin`: только админская часть.
  - `headless`: API/контент без UI.
  - `full`: админ + API + публичный сайт.
  - `playground`: демонстрационный/развёрточный режим.
  - `conformance`: режим для совместимости и проверки контрактов.
2. **Профиль хранения**
  - `memory` — для локального эксперимента.
  - `sqlite` — лучший минимум для небольших инсталляций и локального прод-прототипа.
  - `json-fixtures` — статичные/тестовые фикстуры.
  - `bbolt`, `mysql`, `postgres` — для более устойчивых или боевых сценариев.
  - `browser-indexeddb` — специализированный браузерный сценарий.
3. **Цель запуска**
  - «Только проверить, что работает» → `playground + sqlite`.
  - «Запустить рабочий MVP» → `full + sqlite` (или `mysql/postgres`).
  - «API-first интеграции» → `headless + выбранное хранилище`.

После этих трех шагов вы почти готовы к первому входу.

## 3. Как идёт первый запуск

В GoCMS:

- Конфигурация runtime и профилей инициализируется централизованно через слой bootstrap/preset.
- На старте загружаются выбранные провайдеры и плагины.
- Если провайдеры требуют миграций, выполняются миграции схемы.
- Далее поднимаются сервисы:
  - AuthN/AuthZ
  - Users, Content, Media, Settings
  - Audit и Diagnostics
  - Health registry
  - Snapshot (backup/restore)

### Что проверить после старта

- Сервисы приложения без критических ошибок.
- `Runtime status` в админке показывает базовые health checks.
- Доступ к `/go-login` и `/go-admin`.
- При первом входе — локальный сценарий авторизации/восстановления.

## 4. Первичная аутентификация и учетные записи

GoCMS уже включает hardened local-first механизм:

- local-режим аутентификации с Argon2id-проверкой;
- политика сессии (TTL/таймауты/куки);
- лимитирование неуспешных входов и временная блокировка;
- recovery-коды и административные reset-токены (без email-цепочки);
- API-токены (app passwords) с ограничением срока жизни, ревокацией и маппингом на principal.

### Рекомендуемый onboarding-цикл

Краткий чеклист (подробные шаги, таблицы и перекрёстные ссылки — в **[рекомендуемом цикле первого входа](./onboarding/recommended-first-login-cycle.md)**):

1. Запустите окружение в безопасном режиме.
2. Создайте первого администратора (через bootstrap/инициализацию).
3. Зайдите в `/go-login`.
4. Смените/установите пароль, сохраните recovery-коды вне репозитория.
5. Проверьте разделы Wiki [модель безопасности](./security/security-model-overview.md) и [аутентификация](./authentication/authentication-overview.md) под вашу модель ролей и токенов.

## 5. Роли и capabilities

Управление доступом построено через capability-модель:

- `admin`, `editor`, `viewer` — базовые роли.
- Каждому действию соответствует `Capability`.
- Любое важное действие (в т.ч. админские/privileged API actions) проходит через проверку прав.

Рекомендуемый порядок:

1. Дать минимум прав новым редакторам.
2. Отдельный пароль администратора не должен использоваться для интеграций.
3. Для интеграций использовать app password/API token с привязкой по scope.

## 6. Ключевые сущности CMS (первые шаги в контентной модели)

Работать с GoCMS удобно в порядке:

1. **Content Types** — схема типов данных.
2. **Taxonomies** — структура классификации контента.
3. **Content** — публикация записей по типам.
4. **Media** — медиа-ресурсы и привязка к записям.
5. **Menus** — структура навигации сайта.
6. **Settings** — параметры системы и темы.
7. **Users/Authors** — управление командой.

Сначала заводите минимальный тип контента (например, `page`), затем добавьте несколько полей,
после чего создайте 1–2 тестовые записи и проверьте предпросмотр.

## 7. Публикация и публичный сайт

Для полного цикла:

- Вносим контент.
- Проверяем статус публикации (`draft`/`published` по вашему бизнес-процессу).
- Настраиваем маршруты рендеринга и темы.
- Проверяем публичные endpoints.

Если используете `preview`, настройте отдельный флоу предпросмотра и убедитесь,
что права доступа к предпросмотру не шире, чем нужно.

## 8. Работа с API

Основной интерфейс управления/интеграций — REST:

- `go-json`-эндпоинты для административных и публичных операций.
- Для каждой критичной операции аудитируется actor/action/resource и статус.
- Для интеграций — отдельные API tokens (app passwords), а не административные cookie-сессии.

Также в составе платформы доступен GraphQL-плагин (при активации):

- единый GraphQL endpoint для сложных интеграций и выборочного получения данных.
- при миграции на внешних коннекторах проверьте policy и scopes в настройках интеграции.

## 9. Плагины и расширение

GoCMS проектировался как расширяемый, а не «жёсткий фреймворк».

Типы расширений:

- **Runtime plugin**: регистрация новых routes, settings, capabilities, actions.
- **Admin plugin**: дополнительные экраны/панельные секции.
- **Content/Интеграционный plugin**: domain-specific логика и новые точки доступа.

Базовый набор из коробки уже показывает подход:

- GraphQL plugin;
- JSON import/export plugin (управляемый бэкап/ресто).

Для собственного плагина:

1. Определите контракт (какие routes/возможности/permissions нужны).
2. Реализуйте регистрацию в runtime layer.
3. Пропишите UI/админские экраны в своем блоке/экране.
4. Покройте важные сценарии тестами.

## 10. Эксплуатационные проверки (самый важный блок для 101)

После первого рабочего сценария обязательно проверьте:

### 10.1 Миграции

- Каждое изменение схемы выполняется через versioned миграции.
- История хранится в `schema_migrations`.
- Перед деплоем/свичем хранилища проверьте цепочку и непрерывность `VERSION -> VERSION+1`.

### 10.2 Health checks

- Health registry запускает проверку состояния базы, миграций, snapshot, authn, audit и error-log store.
- Результаты должны быть доступны в Runtime status в админке.

### 10.3 Backup / Restore

- Snapshot — это базовый снимок ядра системы.
- Проводите регулярный экспорт до релизов и перед миграциями.
- При восстановлении проверяйте валидацию пакета и консистентность после импорта.

### 10.4 Audit logs

- Любые привилегированные действия пишутся в аудит.
- Для чувствительных событий проверяйте наличие actor/resource/status и детали.
- Секреты в деталях автоматически редактируются на этапе записи (redaction).

### 10.5 Error logs

- Локальный bounded storage хранит последние ошибки (ограничение по количеству).
- Используйте его для диагностики, когда внешнего центра логирования ещё нет.

## 11. Рекомендуемый порядок первого спринта 1

Если нужно очень быстро выйти в рабочий формат:

1. Поднять систему на `full` + `sqlite`.
2. Создать базовые сущности контента и один тип страницы.
3. Завести одного `admin`, одного `editor`.
4. Включить аудит и проверить, что privileged actions логируются.
5. Настроить миграционный пайплайн и health checks.
6. Сделать первый backup snapshot.
7. Включить автоматическую ротацию error logs/мониторинг runtime status.

## 12. Как не допустить типовых ошибок на старте

- Не запускать production на `memory`.
- Не давать редакторам лишние capabilities без ревизии.
- Не смешивать "playground" и "full" поведение в одном рабочем контуре.
- Не игнорировать предупреждения health-зона до первого релиза.
- Не делать первый релиз без snapshot-плана.

## 14. Что считать готовым после Onboarding 101

- Система запущена и отвечает на `/go-json`, `/go-admin`.
- Созданы и протестированы базовые content-types + публикация.
- У кого есть доступ: минимум `viewer`/`editor`/`admin` с корректными capability.
- Health checks зелёные для критичных компонентов.
- Запущен хотя бы один успешный snapshot.
- Проверен аудит и есть понятный путь диагностики ошибок.

После этого можно переходить к:

1. Доработке темы/UX публичного рендера,
2. Настройке CI/CD для миграций и деплой-пайплайнов,
3. Разработке бизнес-плагинов и расширений API.
