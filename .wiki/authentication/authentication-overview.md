# Обзор аутентификации

Справочник по режимам входа и сопутствующим механизмам (согласован с англоязычной техдокой в каталоге `docs/` репозитория GoCMS; перелинковка между статьями — только внутри [Wiki](../README.md)).

## Политики логина (высокоуровнево)

| Политика | Назначение |
|----------|------------|
| **`local`** | Основной режим для не-демо инсталляций: пользователи в хранилище, Argon2id, recovery-коды, reset-токены, app tokens, учёт попыток входа. |
| **`playground`** | Изолированная песочница / browser-local: упрощённый демо-логин (`admin` / `admin`), не для смешения с боевыми данными. |
| **`external` / `oidc`** | Зарезервировано под внешний IdP; ядро **не** подставляет фикстуры вместо реального провайдера. |

По умолчанию для не-демо сценариев ориентируйтесь на **`local`**.

<a id="local-login"></a>

## Вход `local`

- URL формы входа: **`/go-login`**.
- Идентификатор: **логин или email**.

<a id="playground-login-mode"></a>

## Режим `playground`

Только для изолированного опыта (в т.ч. browser-local). Не используйте те же учётные данные и привычки, что и на **`full`** + **`local`**.

## Сидовые учётные записи (разработка)

При включённом сидировании фикстур в хранилище могут появиться тестовые пользователи с известными паролями (типичный набор описан в англ. `docs/authentication.md`). Для любого контура, выходящего за пределы локального эксперимента, **сразу смените пароли** и пересмотрите выдачу прав.

## Политика сессии

- Cookie-сессии, абсолютный TTL, secure-cookies на прод-подобных deployment-профилях.
- Лимиты попыток входа и блокировка.

Настраиваемые typed settings (имена как в коде/доке):

- `auth.session.idle_ttl_hours`
- `auth.session.absolute_ttl_hours`
- `auth.login.max_attempts`
- `auth.login.attempt_window_minutes`
- `auth.login.lockout_minutes`
- `auth.reset_token.ttl_minutes`

## Действия безопасности в админке (Users)

Из формы быстрого редактирования пользователя администратор может (в **`local`**):

- сменить пароль;
- потребовать смену пароля при следующем входе;
- сгенерировать **recovery-коды**;
- выдать одноразовый **admin reset token**;
- создать **app token** для API;
- отозвать app token по префиксу/ID.

Секреты показываются **один раз** — сохраните их до ухода со страницы.

## Recovery без SMTP

1. **Recovery-коды** (одноразовые, хэш at rest).
2. **Admin reset token** (короткоживущий, одноразовый).

Ядро намеренно не зависит от доставки писем; плагины могут добавить email/OIDC позже.

## App tokens

- Bearer-токены для REST-клиентов, в БД только хэши.
- Поля: префикс/ID, владелец, опциональный scope capabilities, срок, ревокация, last-used.
- Без явного scope наследуются capability от роли пользователя.

Discovery REST может объявлять схему авторизации `app_token` наряду с браузерной сессией.

## Будущий OIDC

Планируемое направление: маппинг claims в локальных пользователей, сохранение локальной модели capabilities, app tokens и аудита внутри GoCMS.
