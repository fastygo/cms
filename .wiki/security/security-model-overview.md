# Обзор модели безопасности

Краткий справочник по тому, как ядро GoCMS относится к секретам, сессиям и аудиту. Согласован с англоязычной техдокой в каталоге `docs/` репозитория; перелинковка между статьями — только внутри [Wiki](../README.md).

## Принципы

- **Local-first**: ядро не требует SMTP, внешнего SaaS-аудита и внешнего IdP по умолчанию.
- Будущий **OIDC** должен подключаться адаптером, не заменяя целиком локальную модель проверки прав.

## Пароли

- Хранение: **Argon2id**, открытый текст пароля нигде не персистится.
- У записи пользователя учитываются **`PasswordUpdatedAt`**, флаг **`MustChangePassword`**, **`LastLoginAt`**.

## Recovery и сброс без почты

- **Recovery-коды** — хэшируются at rest, одноразовые; при генерации показываются один раз оператору.
- **Admin reset token** — хэш, короткий TTL, одноразовый; показ один раз в UI.
- Ядро **не** шлёт ссылки сброса по email — это сознательный выбор для офлайн и автономных инсталляций.

Подробнее сценарии использования: [обзор аутентификации](../authentication/authentication-overview.md).

## App tokens (токены приложений)

- В хранилище лежит только **хэш** токена; для поиска и отзыва хранится **префикс/ID**.
- Область прав: либо явный capability **scope**, либо наследование от ролей владельца.
- Просроченные и отозванные токены отклоняются даже если «сырой» секрет где-то утечёт.

## Сессии браузера

- Подписанные **cookie**-сессии.
- Для прод-подобных **deployment**-профилей включаются **secure**-cookies.
- Есть метаданные **абсолютного** срока жизни сессии.
- Учёт неуспешных входов и **временная блокировка** (пороги настраиваются typed settings — см. [аутентификацию](../authentication/authentication-overview.md)).

## Аудит и редактирование секретов

В аудит пишутся actor, action, resource, статус, метаданные запроса, детали с **редакцией** (redaction): ключи деталей, содержащие `password`, `token`, `secret`, `code`, `hash`, не кладутся в открытом виде.

## Явные non-goals ядра

- Нет доставки сброса пароля через SMTP.
- Нет готового OIDC-логина (пока).
- Нет «удобного» обхода capability-проверок.

## Рекомендации оператору

1. Ротировать **сидовые** пароли при первом «настоящем» использовании.
2. One-time материалы (recovery, reset, app token) сразу переносить в **доверенное офлайн-хранилище**.
3. Для API клиентов использовать **app tokens**, а не админскую cookie-сессию.
4. Экспорты **snapshot** считать чувствительными бэкапами: в них может быть метаданные, нужные для автономного восстановления, включая пользовательскую/auth-связанную информацию.
