# Рекомендуемый цикл первого входа

Краткий чеклист из раздела «Первичная аутентификация» в [Onboarding 101](../onboarding-101.md) — здесь то же самое с пояснениями и перекрёстными ссылками по Wiki.

## 1. Запустите окружение в «безопасном» режиме

Под **безопасным режимом** имеется в виду не один флаг в коде, а связка выбора профиля и данных:


| Цель                        | Рекомендация                                                                                                                                                                                                                |
| --------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Первый осмотр UI, демо      | Профиль `**playground`**: изолированная песочница, отдельная политика входа (см. [обзор аутентификации](../authentication/authentication-overview.md#playground-login-mode)). Не смешивайте этот контур с рабочими данными. |
| Локальная разработка / тест | `**full**` или `**headless**` + хранилище `**sqlite**` (или другое выбранное), `**local**`-политика логина. Не используйте `**memory**` как опору для «почти прода».                                                        |
| Чувствительные данные       | Отдельный инстанс, свой `DataSource`, сильный `SessionKey`, прод-подобный `DeploymentProfile` (влияет на secure-cookies и прочее — см. [модель безопасности](../security/security-model-overview.md)).                      |


После старта проверьте доступ к `/go-login` и `/go-admin` и отсутствие критических ошибок в логах (см. раздел о первом запуске в [Onboarding 101](../onboarding-101.md)).

## 2. Первый администратор: bootstrap и сиды

**Bootstrap** в GoCMS — это выбор хранилища, DSN, миграций и поднятие сервисов при старте процесса (см. [Onboarding 101](../onboarding-101.md)).

Появление **первой учётной записи администратора** зависит от сценария:

- **Сид фикстур** (`SeedFixtures` / типичный dev-запуск): в хранилище создаются предопределённые пользователи с известными паролями **только для разработки**. Их нужно сразу считать компрометированными для любого не-локального контура.
- **Пустая база без сидов**: администратора создаёт процесс инициализации вашего развёртывания (CLI, миграция + ручной create user, или экран первого запуска — в зависимости от того, как собран ваш бинарь/пакет). Уточняйте в документации конкретного дистрибутива GoCMS.

Практическое правило: как только у вас есть первая сессия админа, **не** храните пароли и one-time материалы в репозитории или в общем чате.

## 3. Вход через `/go-login`

1. Откройте в браузере путь `**/go-login`** на базовом URL вашего инстанса.
2. В режиме `**local**` в поле идентификатора допускаются **логин или email** (см. [обзор аутентификации](../authentication/authentication-overview.md#local-login)).
3. В `**playground`** используется упрощённый сценарий входа для песочницы — не переносите эту привычку на prod.

После успешного входа вы обычно попадаете в админ-оболочку (`/go-admin` и связанные маршруты).

## 4. Смена пароля и recovery-коды

1. Сразу **смените пароль** учётки администратора (и любых сидовых пользователей), если инстанс выходит за пределы одноразового локального эксперимента. В админке это делается из раздела пользователей (**Users** / быстрое редактирование).
2. Сгенерируйте **recovery-коды** для критичных учёток. Они показываются **один раз**; ядро хранит только хэши.
3. Сохраните коды **вне репозитория**: менеджер паролей, зашифрованный офлайн-архив, процедура команды — на ваш выбор.

Детали механизмов (без SMTP, reset-токены, app tokens): [обзор аутентификации](../authentication/authentication-overview.md) и [модель безопасности](../security/security-model-overview.md).

## 5. Роли, capabilities и API-токены

Перед выдачей доступа редакторам и интеграциям:

1. Прочитайте [модель безопасности](../security/security-model-overview.md) — пароли, сессии, аудит, redaction.
2. Прочитайте [обзор аутентификации](../authentication/authentication-overview.md) — политика сессии, app tokens, отличие `**local`** от `**playground**`.
3. В [Onboarding 101](../onboarding-101.md) в разделе о ролях кратко описаны `admin` / `editor` / `viewer` и capability-модель: выдавайте минимум прав и **не** используйте пароль администратора для интеграций — заводите **app password / API token** с нужным scope.
