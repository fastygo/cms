## Overview

GoCMS now supports a local-first authentication model for non-demo profiles.

- `playground` keeps the isolated demo login (`admin` / `admin`) for browser-local sandbox use only.
- `local` uses repository-backed users with password hashes, recovery codes, reset tokens, app tokens, and login attempt tracking.
- `external` and future `oidc` policies intentionally do not fall back to fixture credentials.

The default non-demo policy is now `local`.

## Local Password Setup

Fixture seeding creates these local accounts:

- `admin@example.test` with password `admin`
- `mr.gopher@gocms.example.test` (login `mrgopher`) with password `editor`
- `viewer@example.test` with password `viewer`

These credentials are for seeded development/test installs only. Real operators should rotate them immediately from the `Users` screen.

## Login Modes

### `local`

Use the regular `/go-login` form. The identifier may be a login or an email address.

### `playground`

Only for the isolated playground/browser-local experience. This mode keeps the simple `admin` / `admin` flow.

### `external` / `oidc`

Reserved for a future identity-provider adapter. Core does not emulate external auth with fixture fallback.

## Session Policy

Core enforces a local session policy with:

- cookie-based browser sessions
- absolute session expiry metadata
- secure cookies for production-like deployment profiles
- login attempt tracking and temporary lockout

Typed settings reserved for policy tuning:

- `auth.session.idle_ttl_hours`
- `auth.session.absolute_ttl_hours`
- `auth.login.max_attempts`
- `auth.login.attempt_window_minutes`
- `auth.login.lockout_minutes`
- `auth.reset_token.ttl_minutes`

## User Security Actions

From the `Users` quick-edit form, administrators can perform these local-only actions:

- rotate a user's local password
- require a password change on the next sign-in
- generate one-time recovery codes
- issue a one-time admin reset token
- create an app token for API access
- revoke an app token by its stored prefix/ID

Generated recovery codes, reset tokens, and app tokens are shown once. Operators should move them into an offline password manager or another trusted local channel before leaving the page.

## Recovery Without SMTP

Core intentionally does not require email delivery.

Recovery is handled with two autonomous paths:

1. Recovery codes: hashed at rest, one-time use, generated by an administrator.
2. Admin reset tokens: hashed at rest, short-lived, shown once in the admin UI.

This keeps local and offline installs self-contained while preserving a clean boundary for future email- or OIDC-based plugins.

## App Tokens

App tokens are repository-backed bearer tokens for API clients.

- They are stored only as hashes.
- Each token has a lookup prefix/ID, owner user, optional capability scope, expiry, revocation state, and last-used metadata.
- When no explicit scope is supplied, the token inherits the owning user's role-based capabilities.

The REST discovery document now advertises `app_token` alongside browser sessions.

## Future OIDC Path

The local auth layer is now the default source of truth, but it is not coupled to a specific upstream identity provider.

Future OIDC work should:

- map `sub`, `email`, and group/role claims into local users
- preserve local capability resolution
- keep app tokens, audit, and recovery behavior inside GoCMS
- avoid replacing core authn internals with direct IdP assumptions
