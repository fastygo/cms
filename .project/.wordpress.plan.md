# Full WordPress parity

## Реализовать сейчас

- нужны Go-эквиваленты:
    plugin descriptors;
    route/assets/settings/capability registration;
    hook registration + later hook dispatcher;
    content render filters;
    theme assets через manifest;
    editor providers;
    public/admin surfaces.

- **Custom fields / meta API**: metadata есть, но нет полноценной WP-style meta registration, schema, validation, UI, public/private rules по meta keys.
    - registry для meta keys, schema, validation, field UI, ownership/capability rules
    - добавить typed metadata registry и hooks в service layer
- **Media library как в WP**: загрузка бинарников, thumbnails, responsive sizes, image crop/edit, attachment pages, MIME/security policy, storage/CDN providers.
- **Admin list UX**: bulk actions, quick edit, filters, sorting, search, screen options, pagination preferences.
- **Options/settings API**: typed option registry, autoload/public/private behavior, validation, settings groups.
- **Authentication hardening**: real password storage/login through users repository, password reset, app passwords/API tokens, rate limiting.
- **Operational hardening**: migrations, backup/restore, health checks per provider, audit logs, error logs in admin.
- **Pass 7**: финальную public theme

## Реализовать как фичи или дополнения с помощью конфигов или плагинов

- **Фичи**: роли, marketplace, WXR/XML, REST edge cases, comments, advanced revisions UI

- **Scheduling / cron**: статус `scheduled` есть, но нужна cron-like система публикации, background jobs, missed schedule handling.
- **Trash / restore parity по всем сущностям**: для content есть, но не для всех разделов как WP.

## Custom Fields / Meta API
Можно отложить в плагины. Текущий API уже достаточно готов как база:

- В `domain/content.Entry` есть `Metadata map[string]MetaValue`.
- `MetaValue` уже хранит `Value any` и `Public bool`.
- REST/GraphQL уже умеют отдавать public/private metadata с фильтрацией.
- Admin editor уже имеет простейшие `meta_key` / `meta_value`.

Чего пока нет: registry для meta keys, schema, validation, field UI, ownership/capability rules. Это отлично ложится в plugin: `seo-meta`, `acf-like-fields`, `ecommerce-fields`, etc.

Вывод: **ядро готово подхватить Meta API позже без большой переделки**. Возможно понадобится добавить typed metadata registry и hooks в service layer, но модель уже правильная.

## Template Hierarchy
Полный WP-style hierarchy (`single-post`, `taxonomy-news`, `front-page`, etc.) для Go сейчас **не обязателен**.

В Go у нас лучше модель:

- compiled theme package;
- `Theme.Manifest().Templates`;
- `TemplateRole`: `front`, `page`, `post`, `archive`, `taxonomy`, `author`, `search`, `not_found`;
- public renderer передаёт `publicrender.PublicPage` с `Kind`, `Screen`, `TemplateRole`.

То есть Go-тема не обязана искать PHP-файл по цепочке как WordPress. Она может сама внутри `Render()` решить, какой компонент вызвать.

На текущем этапе, где мы обслуживаем данные/контент/метаданные, достаточно текущих `TemplateRole`. Позже можно добавить **optional theme resolver**:

```text
post -> post:{kind}:{slug} -> post:{kind} -> post -> index
taxonomy -> taxonomy:{type}:{term} -> taxonomy:{type} -> taxonomy -> archive
```

Вывод: **не нужно сейчас**. Если понадобится, это лучше сделать как theme capability/config, не как WP-копию в core.

## Localization / PO/MO
Для GoCMS я бы **не уходил в PO/MO-style как обязательный путь**.

Для админки уже есть `adminfixtures`, и это нормально для embedded/admin UI. Для сайтов правильнее:

- content localization через `LocalizedText`;
- locale из request context;
- theme i18n через JSON/YAML/message catalogs;
- optional plugin для translation workflow.

PO/MO полезны для WordPress PHP ecosystem, но в Go они не обязательны. Можно поддержать импорт/экспорт PO/MO позже через plugin, если нужно для совместимости.

Текущий API готов:

- `LocalizedText map[locale]string`;
- public renderer уже использует locale fallback;
- admin/public fixtures можно заменить/расширить catalogs;
- plugin может добавить translation management UI.

Вывод: **PO/MO не нужно в core**. Translation plugin позже возможен без ломки модели.

## WP-Compatible Developer APIs
В точном WP-виде это для Go не нужно.

Не стоит копировать:

- PHP template tags;
- shortcode API как основную модель;
- `wp_enqueue_script` 1:1;
- глобальные actions/filters как в WordPress.

Но нужны Go-эквиваленты:

- plugin descriptors;
- route/assets/settings/capability registration;
- hook registration + later hook dispatcher;
- content render filters;
- theme assets через manifest;
- editor providers;
- public/admin surfaces.

Часть уже есть в `internal/platform/plugins`: manifest, settings, capabilities, hooks registration, assets, routes. Но важная оговорка: **hooks сейчас больше зарегистрированы как metadata**, полноценный execution pipeline для actions/filters ещё нужно будет добавить, когда появится реальный use case.

Вывод: **WP developer APIs не нужны в core 1:1**. Текущая архитектура готова принять Go-native equivalents через plugins, но hook execution/filter pipeline ещё предстоит дозреть.
